Here's exactly how the race plays out. Thread A is compiling the factory for AbsenceRequestV3. Thread B is simultaneously compiling for ClockingRemovalRequestV3. Step one: both threads read the current length of the converter list. The list has five entries. Both threads get the value five — at the same moment, before either has added anything. Step two: both threads bake index five into their generated IL, permanently, at code-generation time. Thread A bakes it in as the slot for the user reference converter. Thread B bakes it in as the slot for the workflow reference converter. Step three: Thread A calls Add first — UserConverter lands at index five. Step four: Thread B calls Add — WorkflowConverter lands at index six. Thread B's factory will now forever look up index five when it expects a workflow reference. But index five holds the user converter. Every call through that factory reads an integer from the database, runs it through the user converter, and when the generated IL tries to cast the result to a workflow reference, it throws an InvalidCastException.
