To bring it together: NPoco's mapping factory uses a static list to store converters and bakes their indices into generated IL — but reading the list length and appending to it are two separate, unprotected operations. Concurrent first-use compilations race on those two steps, producing a factory that permanently applies the wrong converter. The factory is then cached for up to an hour, so the exception repeats on every request for that type until the process restarts. The fix is a five-line patch in our existing NPoco fork: add a private lock object and wrap the read-then-write inside a lock block. That makes the operation atomic, eliminates the race, and protects every NPoco fetch path in the codebase — not just the one call site. The broader lesson: when a third-party library uses static mutable state for first-time initialisation, concurrent startup is a hidden risk. And that risk is at its widest exactly when you're doing a rolling deploy.
