The root cause is inside NPoco's MappingFactory. When it encounters a property that needs a custom converter — for example an IAggregateReference — it stores the converter function in a static, process-wide list called m_Converters. It then emits an ldc.i4 instruction that permanently bakes the converter's list index into the generated IL. The critical flaw is right here: reading the current count and adding to the list are two separate operations. There is no lock, no interlocked operation, nothing. On a single thread this is perfectly fine. Under concurrency, these two steps can interleave between threads in exactly the wrong way.
