The fix is a patch directly in our existing NPoco fork. The change is five lines in NPoco's internal method that builds the converter list. We add a private lock object and wrap the two non-atomic steps - reading the current list length and appending the converter - inside a lock block. That makes the read and write atomic. The factory index baked into the generated code will now always be correct, regardless of how many threads are compiling factories at the same time. This fix protects every NPoco fetch path in the codebase, not just our FetchMultiple call.
