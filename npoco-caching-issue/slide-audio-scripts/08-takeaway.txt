To bring it all together: NPoco's mapping factory uses a static list to store converters and bakes their indices into generated IL — but reading the current count and appending to the list are not atomic. Concurrent first-use compilations for different types can assign the same index to different converters, producing a broken factory that gets cached for up to an hour. The fix is a non-generic process-wide guard class with a semaphore and a dictionary. Double-checked locking ensures first-time compilations are serialised without any overhead in steady state. The change touches two files: BaseRequestStateManager gets a locking wrapper and the hot-path logic extracted into FulfillInternal, and a new NPocoFactoryCompilationGuard class holds the static state. If you want to test it, run concurrent FetchMultiple calls across multiple manager subclasses on a fresh process and assert that no mapping exceptions occur.
