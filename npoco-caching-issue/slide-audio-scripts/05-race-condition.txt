The problem is that this static converter list has no locking whatsoever. Reading its size and appending to it are two completely separate, unprotected operations.

Here is how the race plays out. Thread A is compiling the factory for AbsenceRequest. Thread B is simultaneously compiling for ClockingRequest. Both threads read the current length of the converter list at the same moment - it has five entries, so both threads get the value five. Both bake index five permanently into their generated code: Thread A expecting a user reference converter, Thread B expecting a workflow reference converter. [pause:0.5s] Thread A's Add call lands first - the user reference converter goes to index five. Thread B's Add call follows - the workflow reference converter goes to index six. Thread B now has a factory that will forever look up index five expecting a workflow converter - but index five holds the user converter. Every call through that factory throws an InvalidCastException.

And because that broken factory is immediately stored in NPoco's process-wide memory cache, every subsequent request for that type hits the exact same corrupt factory. The error sticks - it does not self-heal, and it does not go away on retry. The only way to clear it is to restart the process, which wipes the in-memory cache.
