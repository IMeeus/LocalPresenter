The problem is that this static converter list has no locking whatsoever. Reading its size and appending to it are two completely separate, unprotected operations.

Here is how the race plays out. Thread A and Thread B are simultaneously compiling factories. Both read the converter list length at the same moment - say five - and both hardcode index five into their generated code. [pause:0.5s] Thread A adds its converter first, landing at index five. Thread B follows, landing at index six. But the factory generated by Thread B still looks up index five - which now holds the converter added by Thread A. Every call throws an InvalidCastException.

Because NPoco caches that broken factory immediately, every subsequent request hits the same corrupt factory. The error sticks - it does not self-heal and does not go away on retry. The only fix is a process restart.
