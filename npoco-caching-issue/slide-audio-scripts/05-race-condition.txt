The problem is that this static converter list has no locking whatsoever. Reading its size and appending to it are two completely separate, unprotected operations.

Here is how the race plays out. Thread A is compiling the factory for AbsenceRequest. Thread B is simultaneously compiling for ClockingRequest. Both threads read the current length of the converter list at the same moment - it has five entries, so both threads get the value five. Both permanently bake index five into their generated code: Thread A for the user reference converter, Thread B for the workflow reference converter. Then Thread A calls Add first - UserConverter lands at index five. Thread B calls Add - WorkflowConverter lands at index six. Thread B now has a factory that will forever look up index five when it expects a workflow converter - but index five holds the user converter. Every call through that factory throws an InvalidCastException.

And because that broken factory is immediately stored in NPoco's process-wide memory cache, every subsequent request for that type hits the exact same corrupt factory. The error is indefinitely persistent - it does not self-heal, and it does not go away on retry. The only way to clear it is to restart the process, which wipes the in-memory cache.
