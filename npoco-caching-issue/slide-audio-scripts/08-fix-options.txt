There are four approaches on the table. Option one is a static mutex in Fulfill - a process-wide guard class that serialises factory compilation. It's the lowest-risk change with no new dependencies, and it adds zero overhead once all factories are compiled. Option two is startup pre-warming: compile all factories in a single-threaded hosted service before the web server starts accepting connections. No runtime overhead at all, but it requires enumerating all managers via dependency injection. Option three - replacing FetchMultiple with three separate Fetch calls - doesn't actually fix the race because the same static list is involved in all factory compilations, and it adds extra database round-trips on top of that. Not recommended as a standalone fix. Option four is forking NPoco and adding a five-line lock around the problematic section. It's the most correct fix, fixes the root cause directly, and since we already have a fork it's the recommended approach.
