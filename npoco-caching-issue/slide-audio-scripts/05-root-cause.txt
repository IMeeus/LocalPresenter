NPoco's MappingFactory is responsible for mapping database query results to dot net objects. For simple value types this is straightforward, but for complex types — like aggregate references to users or workflows — it generates factory delegates at runtime using intermediate language emission. Each factory needs to know which converter function to use for each property, so it stores those converter functions in a static, process-wide list and permanently bakes the converter's index in that list into the generated code. That index is baked in permanently — it never changes.

The critical flaw is right here: reading the current list length and appending to it are two separate operations, and there is no lock anywhere in this path. On a single thread this is fine. Under concurrency, these two steps can interleave between threads in exactly the wrong way.

Here's exactly how the race plays out. Thread A is compiling the factory for AbsenceRequest. Thread B is simultaneously compiling for ClockingRequest. Both threads read the current length of the converter list at the same moment — it has five entries, so both get the value five. Both threads permanently bake index five into their generated intermediate language: Thread A for the user reference converter, Thread B for the workflow reference converter. Then Thread A calls Add first — UserConverter lands at index five. Thread B calls Add — WorkflowConverter lands at index six. Thread B now has a factory that will forever look up index five when it expects a workflow reference. But index five holds the user converter. Every call through that factory reads an integer from the database, runs it through the user converter, and when the intermediate language tries to cast the result to a workflow reference, it throws an InvalidCastException.
