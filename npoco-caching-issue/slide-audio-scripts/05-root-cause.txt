The root cause is inside NPoco's MappingFactory. When it encounters a property that needs a custom converter - for example a reference to a user or a workflow - it stores the converter function in a static, process-wide list. It then emits an IL instruction that permanently bakes the converter's position in that list into the generated factory code. The critical flaw is right here: reading the current list length and appending to it are two separate operations. There is no lock, no atomic operation, nothing. On a single thread this is perfectly fine. Under concurrency, these two steps can interleave between threads in exactly the wrong way.
